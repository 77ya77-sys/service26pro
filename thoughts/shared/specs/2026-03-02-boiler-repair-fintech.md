# Landing Page: Ремонт газовых котлов (Финансовый стиль) Specification

## Executive Summary
Высококонверсионный одностраничный сайт для сервисного центра по ремонту и обслуживанию газовых котлов. Цель: мгновенный захват заявок ("горячего" трафика от паникующих клиентов и запросов на плановое ТО) через прямые звонки и WhatsApp. Упаковка продукта строится на утилитарном контенте внутри строгого и премиального «финтех» UI.

## Problem Statement
Клиенту, столкнувшемуся с поломкой котла (особенно в холодное время года), нужно максимально быстрое, надежное решение. Проблема большинства конкурентов — перегруженные, устаревшие сайты или шаблонные блоки, не дающие уверенности в экспертности мастера. Бизнесу требуется получать заявки моментально, без сложного заполнения форм клиентом. Решение: чистый "Finpay-стиль", где каждый элемент (социальные доказательства, бенто-структура, мгновенные CTA) работает на доверие и скорость.

## Success Criteria
1. Высокая скорость загрузки первого экрана и работы анимаций.
2. Безошибочная работа динамической подмены заголовков на основе UTM-меток.
3. Доставка лидов напрямую через звонки и мессенджеры без промежуточных баз данных.
4. Отличная читаемость (контраст ≥ 4.5:1) и строгая сетка на мобильных устроиствах.

## User Personas
1. **Паникующий клиент (Экстренный ремонт)** — У него холодно, котел шумит или выдает ошибку. Ему не нужны долгие чтения, ему нужна гарантия результата ("Выезд за 30 минут") и кнопка быстрого звонка или перехода в WhatsApp.
2. **Плановый клиент (Обслуживание)** — Ищет профилактику. Готов изучать "Как это работает", прайс-лист в бенто-карточках и кейсы (блок доверия).

## User Journey
1. Переход из контекстной рекламы или SEO по конкретному запросу (например, "обслуживание котлов").
2. Скрипт мгновенно считывает UTM-метку и подстраивает Hero-заголовок ("Сломался котел... " или "Плановое ТО перед сезоном...").
3. Клиент скроллит чистый интерфейс, где блоки мягко появляются через (fade-in-up).
4. Клиент находит свою проблему в Блоке 2.
5. Нажатие на любую кнопку CTA ("Вызвать мастера" или форма в Блоке 8) не открывает сложную форму, а сразу инициирует звонок или открывает чат WhatsApp с предзаполненным текстом (например: "Здравствуйте, сломался котел...").
6. На мобильной версии клиент может в любой момент нажать на закрепленную кнопку звонка или иконку WhatsApp на экране (Sticky элемент).

## Functional Requirements
### Must Have (P0)
- **Динамическая подмена контента**: JS-скрипт, парсящий строку URL (например, `?utm_term=to`) и подменяющий заголовок Hero-блока.
- **Интеграция с WhatsApp**: Все элементы сбора лидов (Включая Блок 8) формируют специальную ссылку API WhatsApp `wa.me/Номер?text=...`.
- **Sticky CTA**: На мобильной версии закрепленная кнопка с прямым звонком курьером (`href="tel:..."`) и иконка WhatsApp.
- **Бенто-сетка (CSS Grid)**: Жесткая модульная структура карточек для блоков: Аудитория (Блок 2), Выгоды (Блок 3), Прайс (Блок 5), Social proof (Блок 6).
- **Соблюдение UI-правил (Finplay-стиль)**: Строгий гротеск, акцентный цвет только на CTA и цифрах, одинаковые скругления углов, легкие тени, контраст текста.

### Should Have (P1)
- **Функциональная анимация**: Плавное появление (fade-in-up) блоков при скролле через Intersection Observer (без тяжелых библиотек). Длительность 0.3-0.4s.
- **Аккордеоны FAQ**: Тонкие линии и плавная функциональная анимация раскрытия на чистом CSS/JS.
- **Смена медиа**: Если заголовок подменяется для "ТО", визуализация (фото) в Hero также может быть заменена для релевантности (опционально).

### Nice to Have (P2)
- Автоматическое определение мобильных устроиств для оптимизации отображения Sticky-элементов.

## Technical Architecture
### Data Model & System Components
Всё приложение является статичным фронтенд-решением (HTML/CSS/JS), не требующим подключения баз данных или сложного бэкенда.
- **HTML5**: Семантическая структура, разбитая на 8 функциональных блоков.
- **CSS3 (Vanilla)**: Стилизация, CSS Variables для цветовой схемы и типографики, CSS Grid для формирования бенто-сеток. Мобильная адаптация через `@media` (Mobile-first подход).
- **Vanilla JS**:
  - `utm-parser.js`: Модуль чтения GET-параметров URL и замены DOM-узлов (заголовок H1/картинки).
  - `scroll-observer.js`: Модуль, использующий `IntersectionObserver` для добавления классов видимости (анимация появления).
  - `faq.js`: Модуль управления состоянием аккордеонов.
  - `contact-handler.js`: Формирование ссылок WhatsApp в зависимости от точки входа.

### Integrations
- Прямая интеграция с телефонными приложениями (через `tel:` URI scheme).
- Прямая интеграция с WhatsApp API (через `wa.me/` URI scheme).
- Хостинг: Обычный shared-хостинг (PHP доступен, но не требуется для текущей реализации контактов через мессенджеры).

## Non-Functional Requirements
- **Performance**: Отказ от тяжелых фреймворков и библиотек анимаций (GSAP и тд) в пользу Vanilla JS и CSS. Быстрая загрузка критична для паникующей аудитории.
- **Responsiveness**: Идеальное отображение на всех мобильных устройствах (проектирование от зоны большого пальца).

## Out of Scope
- Сложные многошаговые формы (Квизы).
- Сохранение заявок в базу данных MySQL.
- Подключение Telegram-ботов для отправки заявок.
- Система управления контентом (CMS / админка).
- Фальшивые текстовые отзывы (только фото/лого).

## Open Questions for Implementation
- Какие именно UTM ключи будут использоваться вашим маркетологом для подмены (например, `?utm_campaign=to` или `?utm_term=obsluzhivanie`)? *Для разработки заложим базовые `?utm_source=...`.*
- Какой конкретный контрастный акцентный цвет (НЕ красный) использовать для кнопок, чтобы он гармонировал с "финтех" стилем Finpay?

## Appendix: Research Findings (UTM Content Swapping)
В ходе обсуждения выбрана механика подмены заголовков на стороне клиента (через JavaScript и чтение `window.location.search`), а не на стороне сервера (создание нескольких копий `index.html`). Это упростит масштабирование рекламных кампаний — вам достаточно будет просто генерировать нужную ссылку в кабинете Яндекс.Директа.
